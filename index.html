Quartz2D详解

发表于 2016-02-25   |   分类于 iOS   |     |   阅读次数

什么是Quartz2D?

简单来说，Quartz2D是用C语言封装的二维绘图引擎，同时支持iOS和Mac系统，利用Quartz2D，我们可以完成以下工作：

  • 绘制图形 : 线条\三角形\矩形\圆\弧等
  • 绘制文字
  • 绘制\生成图片(图像)
  • 读取\生成PDF
  • 截图\裁剪图片
  • 自定义UI控件

学习Quartz2D的价值在哪里

虽然Quartz2D属于偏底层的API，并且UIKit已经把我们经常使用的控件封装好，但是，面对一些自定义控件，复杂的控件，UIKit就显得有些无力了。

学好Quartz2D，我们就可以：

  • 绘制一些系统UIKit框架中不好展示的内容，例如饼图，环形进度条，统计图表
  • 自定义一些控件
  • 不添加UI控件的情况下，使UI内容更丰富
  • ……

在iOS中，我们所看见的大部分控件都是利用Quartz2D绘制出来的。

什么是图形上下文

简单来说，图形上下文就相当于一块画布，我们可以在这块画布上画任何我们想要的内容，最后展示在View上。

然而，这个画布一定要在drawRect:方法里面去获取，实现才有效，下面就具体讲下这个方法。

drawRect:

为什么要实现drawRect:方法才能绘图到view上？

因为在drawRect:方法中才能取得跟view相关联的图形上下文

在drawRect:方法中取得上下文后，就可以绘制东西到view上

drawRect如何绘图到View上

View内部有个layer（图层）属性，drawRect:方法中取得的是一个Layer Graphics Context，因此，绘制的东西其实是绘制到view的layer上去了

View之所以能显示东西，完全是因为它内部的layer

drawRect:方法如何调用的

  • 当view第一次显示到屏幕上时，系统会创建好一个跟当前view相关的Layer上下文
  • 系统会通过此上下文，在drawRect:方法中绘制好当前view的内容
  • 主动让view重绘内容的时候，调用setNeedsDisplay或者setNeedsDisplayInRect:。我们主动调用drawRect:方法是无效的。
  • 调用view的setNeedsDisplay或者setNeedsDisplayInRect:时
  • 注意：setNeedsDisplay和setNeedsDisplayInRect:方法调用后，屏幕并不是立即刷新，而是会在下一次刷新屏幕的时候把绘制的内容显示出来。

也正是系统会在调用这个方法之前创建一个与该view相关的上下文，才让我们可以在drawRect:方法中绘制。

注意：在其他地方拿不到view相关的上下文，所以不能实现绘制。

自定义View的步骤

如何利用Quartz2D绘制东西到view上？

  1. 新建一个类，继承自UIView
  2. 实现- (void)drawRect:(CGRect)rect方法，然后在这个方法中
  3. 取得跟当前view相关联的图形上下文
  4. 绘制相应的图形内容
  5. 利用图形上下文将绘制的所有内容渲染显示到view上面

MVVM+ReactiveCocoa简单介绍

发表于 2016-02-23   |   分类于 iOS   |     |   阅读次数

什么是MVVM

MVVM的由来

MVVM是Model-View-ViewModel的简写。微软的WPF带来了新的技术体验，如Silverlight、音频、视频、3D、动画……，这导致了软件UI层更加细节化、可定制化。同时，在技术层面，WPF也带来了 诸如Binding、Dependency Property、Routed Events、Command、DataTemplate、ControlTemplate等新特性。MVVM（Model-View-ViewModel）框架的由来便是MVP（Model-View-Presenter）模式与WPF结合的应用方式时发展演变过来的一种新型架构框架。它立足于原有MVP框架并且把WPF的新特性糅合进去，以应对客户日益复杂的需求变化。

MVVM的优点

MVVM模式和MVC模式一样，主要目的是分离视图（View）和模型（Model），有几大优点:

  1. 低耦合。视图（View）可以独立于Model变化和修改，一个ViewModel可以绑定到不同的”View”上，当View变化的时候Model可以不变，当Model变化的时候View也可以不变。
  2. 可重用性。你可以把一些视图逻辑放在一个ViewModel里面，让很多view重用这段视图逻辑。
  3. 独立开发。开发人员可以专注于业务逻辑和数据的开发（ViewModel），设计人员可以专注于页面设计，使用Expression Blend可以很容易设计界面并生成xaml代码。
  4. 可测试。界面素来是比较难于测试的，而现在测试可以针对ViewModel来写。

MVVM相比MVC好在哪里

与MVC类似，MVVM实际上就是在MVC的基础上进行了又一次的优化，我们知道传统MVC模式主要分为Model，View，Controller三层，Model负责数据层，Controller负责业务逻辑层，View负责界面显示层，所以可以让不同的View使用用一个Controller层，程序扩展性比较好。所以我就不必多说这个模式了好处了。

但是，这个模式并不是完美的，MVC的核心是Controller，它负责与Model通讯，从中取数据，还要将Model取出来的数据进行一定的处理，转化成View层可以直接被使用的数据，如果View层发生了变化，Controller还要负责更新Model层，而Model其实并不与View直接产生交互，它们之间只能通过Controller去获得联系。

如此一来，Controller的任务就略显繁重了，当一个Controller类中，有近1000行代码时，我认为，别人看这个代码，内心应该是奔溃的。太多的逻辑放在了Controller里，这对于后期维护，是很不利的，当你需要在几百个方法里找更新UI的代码时，你很可能会找不到。

而MVVM得出现就是为了简化Controller的代码，把本该不属于Controller的任务分发出去，交给别人去处理，比如网络请求，数据解析，界面内容逻辑，把它们单独放在一个类里面。这样Controller就变得不那么臃肿了。由于去除了大部分的界面逻辑，MVVM可以进行测试了，这也是MVC不敢想的。

MVVM的3大模块

接下来进入正题，看看MVVM设计模式是如何解决这些问题的。
MVVM = Model + View + ViewModel
看上去和MVC差不多，就是把ViewController替换成了ViewModel。如果是这样的话，MVVM压根就没有存在的必要了。所以并不是进行了简单的替换。MVVM由3部分组成：Model ，View ，ViewModel。

  • Model: 作为数据的容器
  • View: 负责界面的展示以及用户交互的处理
  • ViewModel: 负责业务逻辑处理

但实际上，iOS开发是不可能完全抛弃Controller，没有Controller，很多功能是无法实现的，实际上iOS的MVVM模式基本是这样的：

  • Model: 作为数据的容器
  • View: 负责界面的展示以及用户交互的处理
  • ViewController: 用代码创建视图；胶水代码，连接View和ViewModel
  • ViewModel: 负责业务逻辑处理

iOS的MVVM实际上是在原有MVC的基础上，额外增加了一个新的模块ViewModel，用来处理主要业务逻辑，而Controller就只负责连接他们三者，不再负责业务逻辑，这样我们可以把逻辑和UI分的更加清楚，而不是MVC那样全部在一起。

那么，我们可以理解为，iOS的MVVM实际上是MVC的加强版

MVVM之间是如何通讯的

由于MVVM把业务逻辑几乎都放在了ViewModel层去处理，Model层不再和ViewController直接打交道。和Model层直接打交道的是ViewModel层。

简单来说，ViewController通过调用ViewModel的实例方法，来获得数据。一般ViewController会包含一个ViewModel的对象，ViewModel会封装好网络请求，数据处理等一系列方法。当ViewModel从Model取得数据后通过block回调，或者KVO监听属性变化来通知ViewController刷新UI。这就是简单的MVVM的通讯方式。如果更复杂的，用户需要通过View层修改Model，同样也要通过ViewModel的实例方法去实现。

然而不管是block还是代理通知，MVVM大量使用block甚至是KVO都不是一个最好的决策。过多的通知代理，管理起来就显得比较麻烦了，那么有没有更好的方法呢？

ReactiveCocoa

ReactiveCocoa的出现，很好的解决的MVVM中block通知代理过多，管理混乱的问题，ReactiveCocoa为事件定义了一个标准接口，从而可以使用一些基本工具来更容易的连接、过滤和组合。

什么是ReactiveCocoa

ReactiveCocoa（简称为RAC）,是由Github开源的一个应用于iOS和OS开发的新框架,在我们iOS开发过程中，当某些事件响应的时候，需要处理某些业务逻辑,这些事件都用不同的方式来处理。比如按钮的点击使用action，ScrollView滚动使用delegate，属性值改变使用KVO等系统提供的方式。其实这些事件，都可以通过RAC处理
ReactiveCocoa为事件提供了很多处理方法，而且利用RAC处理事件很方便，可以把要处理的事情，和监听的事情的代码放在一起，这样非常方便我们管理，就不需要跳到对应的方法里。非常符合我们开发中高聚合，低耦合的思想。

ReactiveCocoa结合了几种编程风格：

  • 函数式编程（Functional Programming）：使用高阶函数，例如函数用其他函数作为参数。
  • 响应式编程（Reactive Programming）：关注于数据流和变化传播。

简单来说，使用ReactiveCocoa，可以很方便的把MVVM中的View，ViewModel，Model连接起来，通过一系列绑定，实现相应式的架构，彻底摆脱繁杂难于管理的KVO方式。而ReactiveCocoa独特的事件流的编程方式，是开发思路更加清晰，编程更加方便。

采用ReactiveCocoa存在的问题

好处说了这么多，大家一定心动了吧，但是，采用ReactiveCocoa编程也会存在一些问题。

  1. 与别的第三方一样，ReactiveCocoa也属于第三方框架，在开发中我们不能太依赖于某个框架，否则这个框架不更新了，导致项目后期没办法维护，比如之前Facebook提供的Three20框架，在当时也是神器，但是后来不更新了，也就没什么人用了。
  2. ReactiveCocoa的学习成本较高，需要一定的时间与精力。
  3. 采用ReactiveCocoa编程的项目，对于不会ReactiveCocoa的工程师来说，维护较为困难。它们可能看不懂你写的是什么。
  4. ReactiveCocoa对于多人合作的项目，需要更默契的配合和沟通，否则维护成本更大。

互联网？新一轮的机遇？

发表于 2016-01-17   |   分类于 杂谈   |     |   阅读次数

　　说到互联网，大家一定不会陌生，从我们开始接触电脑开始，这个词就一直陪伴这我们，可以说，社会发展到今天，互联网发挥着越来越重要的作用。
　　然而，如今的互联网的发展正朝着移动设备推进，在10年以前，你能想到现在的人们可以在手机上看球赛，刷微博，甚至分享自己的生活么。正是因为移动设备的便捷性，使得互联网再次焕发了生机。

移动互联网的诞生

　　移动互联网的诞生可以说是多方面的共同促进的。没有其中一个，移动互联就不会变的现在这么流行与火爆。
　　首先，是通讯技术的进步。３Ｇ网络的诞生，使得我们手机可以像电脑一样上网，不需要再连网线，不用再为手机网速慢而急躁不安。这一切，都归功于通讯技术的飞速发展，有了高速的移动数据传输，为移动互联网的发展奠定了基础。
　　其次是手机这类移动终端的兴起，移动设备作为互联网传播的载体，在互联网发展中的重要性不言而喻。从乔布斯发布第一款iPhone开始，移动设备以它体积小，使用便捷，随身携带的绝对优势，被越来越多的人当做获取信息的主要载体。移动设备发展到今天，无论是从娱乐性，还是工作上已经能和传统PC相媲美。手机，在今天，已经不仅仅只是用来打电话，发短信了，它成为了我们生活中不可缺少的一部分。在家里，我们可以用手机看电视，看报纸，在商场，我们可以用手机买衣服，在公园，我们可以用手机拍照，录旅行游记。正是因为手机这个移动设备的流行，互联网通过这个载体，使得信息的传播更加快速，与准确。
　　最后，就是支撑手机等移动终端的移动操作系统的诞生。如果说手机是移动互联网的载体，那么，移动操作系统和软件就是移动互联网的魂，一个再好的设备，没有好的用户交互，没有很棒的应用，是不会火的。然而以苹果为首的iOS系统和谷歌的Android系统的出现，使得手机变的和电脑一般的智能，基于操作系统所衍生出来的第三方软件市场更是加快了手机智能化的步伐。通过安装APP的方式来扩展手机的功能，已经成为了主流，正是这种类似电脑的强大的操作系统，成就了移动互联网的蓬勃发展。如今手机的APP已经渗透到了生活的方方面面，我们的生活似乎已经离不开它们了。

移动互联网的从业人员

　　互联网这个圈子很大，里面的从业人员，从技术，人事，再到运营，可以说面面俱到，包含了社会各个层面的人才。互联网不同于传统产业，需要大量的前期投入，互联网本身并不是产品，互联网是一个平台，为传统产业提供技术，人才，投资市场，等等一切它们所需要的服务。互联网是一个高端人才聚集的行业，互联网依赖网络技术，这就决定了，互联网需要的人才需要懂一定的网络技术，要对计算机有所了解。如果是技术团队的人才，你甚至还要懂很多编程的知识，这在一定上也决定了，这行必须是由高端人才，他们对未来充满激情，头脑聪明，擅长学习新的技术的年轻一代引领。
　　我在互联网也算是工作了小半年，虽然接触的圈子非常有限，大多是技术团队，从中大概总结了下从事互联网的的一般特征：

  • 大多年轻，80后、90后居多
  • 对新鲜的技术比较敏感，喜欢新的东西
  • 大多集中在较发达城市，人员流动性大

　　由于年轻人较多，他们对新的事物，没有老一代人那么排斥，大多数都是主动接受，80后，90后本生就是社会的未来，他们的价值观决定了未来的中国的走向。他们不喜欢被拘束，被管教，所以，不同于传统行业，互联网企业对员工的管理也是非常的人性化。很多创业型互联网公司，创始人本身就是80后90后，他们更清楚员工该怎么管理，这也是互联网公司不同于一般企业的地方。

移动互联网的盈利模式

　　移动互联网的盈利模式和传统互联网的盈利模式，在我个人看来其实并没有什么太大的区别。我们知道，传统互联网公司的收入一般来自广告，平台推广，排名竞价。其中，广告投放应该算是互联网收入的主要来源，因为互联网传播的高效，广告推广往往比传统媒体来的更加有效，成本也会相对低廉。随着这两年O2O的兴起，移动凭借其独有的优势，巧妙得将线下商家和互联网联系了起来。
　　然而，移动互联网发展到了今天，绝大多数公司并没有实现盈利。相反，亏损是一年比一年加剧，这种靠补贴的推广，前期投入非常大。许多创业公司往往撑不到一年就出现投资资金链断裂，而无力继续经营下去。而那些有前途的公司，如滴滴，美团，虽然每年亏损更多，但是它们可以吸引比投入多几倍的投资，其中不乏bat的参与。它们在几年内能迅速发展壮大，也要感谢这些参与融资的公司。尽管如此，这种靠融资补贴的O2O能火多久呢？一旦投资商撤资，对于这类公司的打击将是毁灭性的。
　　那么，我们再看看国内互联网３巨头，百度，阿里，腾讯是怎么盈利的。百度的收入绝大多数都是来自搜索引擎的广告收入和竞价收入，由于百度在搜索领域的地位，光广告收入就足够百度运营。阿里巴巴是电商的领军者，它主要靠平台加盟，金融交易，小部分广告。你的每一笔在阿里巴巴的交易，阿里巴巴都能分到一杯羹，也许这点对于我们真的只是九牛一毛，但是，随着用户基数上来，这收入也是可观的。腾讯现在拥有全中国最多的用户量，微信近５亿的用户，再加上微信这种大家都离不开的产品，推广对于腾讯几乎就是零成本。当然广告还远远不够，腾讯的收入很多还来自游戏运营。没错，做游戏是很赚钱的，仗着用户多的优势，腾讯现在在游戏领域也算是国内的No1了。
　　通过分析，我们也许会发现，这三家公司都有一个共同的特点，那就是用户数量已经达到了惊人的数量，这也是互联网的关键。抢到更多的用户，就能在未来的几年占据优势。那么，我们大概就可以理解现在移动互联网为什么要补贴式的推广了，就是抢用户数量。一旦用户数量起来，广告投放，都是轻而易举的事情。

机遇和挑战

　　俗话说的好，不喜欢当官的士兵不是好士兵。我相信每个人心中都有一个伟大的梦想，不管它是多么的天马行空，多么的不靠谱。而互联网恰恰是最容易让你的梦想实现的地方。
　　我们这一代人，都喜欢做自己喜欢的事，随心所欲。我相信大多数人都不甘心一辈子做员工吧，在如今互联网大火的形势下，大家都想搏一把，万一成功了呢？机会是不等人的，抓住了机遇，你或许就能成功。
　　当然，创业的路并不是那么简单，你可能会面临资金问题，员工问题。创业之路不可能是一帆风顺的，你会面临许多挑战，你甚至会想到放弃创业。但是，只要你努力过，就一定会有收获，并且会受益终生。

题外话

　　扯了这么多，我也挺累的，作为一名程序猿，很多时候想的最多只是怎么解决ｂｕｇ而已，创业说句实话，想都不敢想。。。那为什么还要提呢？现在不敢想，不代表１０年之后不敢想，每个人都有梦想，不要让自己的梦被现实所淹没。
　　我还是相信，努力总会有回报，相信自己，相信未来，相信移动互联网。

Markdown 的语法的简要规则

1.标题

标题是每篇文章都需要也是最常用的格式，在Markdown中，如果一段文字被定义为标题，只要在这段文字前加 # 号即可。

# 一级标题

## 二级标题

### 三级标题

#### 四级标题

##### 五级标题

###### 六级标题

总共六级标题，建议在 # 号后面加一个空格，这是最标准的 Markdown 语法。

2.列表

熟悉 HTML 的同学肯定知道有序列表与无序列表的区别，在 Markdown 下，列表的显示只需要在文字前加上 - 或 * 即可变为无序列表，有序列表则直接在文字前加1. 2. 3. 符号要和文字之间加上一个字符的空格。

3.引用

只需要在文本前加入 > 这种尖括号即可。

> 你的内容

4.图片与链接

插入链接与图片的语法很像，区别在一个 ！ 号

图片为：！[](){ImgCap}{/ImgCap}

链接为： []()

5.粗体与斜体

Markdown的粗体和斜体也非常简单，用两个 *包含一段文本就是粗体的语法，用一个 *包含一段文本就是斜体的语法。

例如：我是粗体 我是斜体

6.表格

表格的比较麻烦，例子如下：

| 姓名        | 年龄     |  得分 |
| ---------- | :------:|------:|
| 小张        |26       |98     |
|小李         |34       |95     |

效果如下

姓名 年龄 得分
小张 26 98
小李 34 95

使用FMDB实现iOS数据库增删改查

什么是FMDB

大家都知道，iOS系统使用的是嵌入式数据库sqlite实现数据的管理，并且iOS的官方已经把API写好，作为开发者，我们只需要调用API就能实现数据库的建立，增删改查等功能。

但是，官方给的API是用C语言写的，用起来总感觉有点不方便，并且官方的API也没有针对多线程进行优化。也就是说，如果你使用原生的API建立数据库，又不进行多线程保护，就会产生数据问题。这对于一个APP是致命的，于是针对官方sqlite的API封装的开源第三方库应运而生。

其中最好用的莫过于FMDB了，那么他到底好在哪里呢，下面就让我罗列给大家：

  • FMDB 是基于iOS sqlite3框架的更高一层的封装，API是用Objective-C写的，代码上相比C语言的API更加好用，使用起来更加面向对象，省去了很多麻烦、冗余的C语言代码。
  • FMDB 对比苹果自带的Core Data框架，更加轻量级和灵活。
  • FMDB 提供了多线程安全的数据库操作方法，有效地防止数据混乱。

FMDB的三个核心类

  1. FMDatabase：一个FMDatabase对象就代表一个单独的SQLite数据库，用来执行SQL语句。

  2. FMResultSet：使用FMDatabase执行查询后的结果集。

  3. FMDatabaseQueue：用于在多线程中执行多个查询或更新，它是线程安全的。
